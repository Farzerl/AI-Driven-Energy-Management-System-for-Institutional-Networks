# Team Capability and Operational Ownership

The public repository records responsibilities rather than personal contact details. Team names, student details, and official contacts belong in the proposal and submission form.

## Development responsibilities

| Responsibility | Required capability |
|---|---|
| Power systems and safety | Electrical network understanding, load classification, tariff interpretation, protection boundary |
| AI and data | Data quality, chronological validation, forecasting, classification, baseline comparison |
| Backend and dashboard | Python, FastAPI, browser interface, logging, tests, packaging |
| Edge and metering | Raspberry Pi or gateway operation, CSV/API/Modbus integration, buffering, health monitoring |
| Evidence and submission | Proposal alignment, claim control, documentation, demo, repository hygiene |

## Pilot ownership

- **Estates and Facilities:** operational review and action approval.
- **Electrical and Electronic Engineering:** model, gateway, and technical support.
- **Finance:** private tariff and bill verification.
- **ICT or approved hosting team:** server, access, backup, and network controls.
- **Project team:** configuration, training, incident support, and evidence review.

## Support pathway

The early pilot should name:

- one operational owner;
- one engineering administrator;
- one support contact;
- one Finance verifier;
- one escalation route for meter, network, or safety incidents.

## Knowledge-transfer requirement

The system must not depend on unexplained generated code. Before pilot handover, the team must be able to explain:

- data schema and quality flags;
- forecast target and validation;
- rule and AI trade-offs;
- API endpoints;
- edge buffering;
- security controls;
- cost-model assumptions;
- deployment and rollback.
